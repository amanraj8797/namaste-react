npm:

npm (Node Package Manager) is a package registry + package manager for JavaScript and Node.js. It lets developers:

Publish packages (libraries, tools)

Install dependencies in projects

Manage versions of packages

=> Example: npm install react

This downloads the React package from the npm registry into your project.

=> CDN 

A CDN (Content Delivery Network) hosts files and serves them directly to browsers via URLs across global servers.

- Yes — you need an active internet connection to use CDN links in your project.

Example CDNs: Cloudflare, Akamai, jsDelivr, unpkg

Example CDN usage: <script src="https://cdn.jsdelivr.net/npm/react@18/umd/react.production.min.js"></script>

Here jsDelivr CDN is serving a package that originally comes from the npm registry.

During development, using CDN is convenient.

For production apps, many developers prefer local copies or bundlers (like Parcel/Webpack) to avoid internet dependency and ensure stability.

=> simple comparision

| Feature      | npm                     | CDN                |
| ------------ | ----------------------- | ------------------ |
| Purpose      | Package management      | File delivery      |
| Used by      | Developers (build time) | Browsers (runtime) |
| Installation | `npm install`           | `<script src="">`  |
| Example      | install React           | load React via CDN |


"IMP" ==>> A CDN like jsDelivr or unpkg can serve files from npm packages, but npm itself is not a CDN.

====>> tilde (~) and caret (^)

In npm or Semantic Versioning, tilde (~) and caret (^) are used in package.json to control which versions of a dependency are allowed to update automatically.

# Caret (^): The caret allows updates that do NOT change the major version.

EXAMPLE: "react": "^18.2.0" means ( >=18.2.0  <19.0.0 )

# Tilde (~): The tilde allows only patch updates (bug fixes).

Example: "react": "~18.2.0" means >=18.2.0  <18.3.0


| Symbol | Updates allowed | Example           |
| ------ | --------------- | ----------------- |
| `^`    | minor + patch   | `18.2.0 → 18.9.0` |
| `~`    | patch only      | `18.2.0 → 18.2.5` |
| none   | exact version   | only `18.2.0`     |


=> MAJOR.MINOR.PATCH

18.2.1
│  │  │
│  │  └ bug fixes
│  └ new features
└ breaking changes

Easy way to remember:

^ → more flexible updates

~ → safer updates

============= BUNDLER
What is a Bundler in Development?

A bundler is a tool that combines many files (JS, CSS, images, modules) into a smaller number of optimized files that the browser can load efficiently.

Modern JavaScript apps use many modules like: components, utility functions, CSS files, images.

Browsers would have to load hundreds of files, which is slow. A bundler merges them into bundles.

 Example structure before bundling: 
src/
 ├─ index.js
 ├─ utils.js
 ├─ components/
 │   └─ Button.js
 └─ styles.css

After bundling: 
dist/
 └─ bundle.js


Now the browser loads one file instead of many 🚀

Common bundlers: Webpack, Parcel, Vite, Rollup

# Parcel Bundler

Parcel is a zero-configuration bundler for web apps.

That means you don't need to configure anything to start.

Install Parcel:
npm install parcel --save-dev
~ npm install D parcel
OR
npm install -g parcel (globally)

Basic Example:

Project:

project/
 ├─ index.html
 ├─ index.js
 └─ package.json

Run Parcel:

parcel index.html

Parcel will: 
Start a development server,
Bundle JS, CSS, assets,
Enable hot reload,
Optimize code automatically

## package.json, package-lock.json, node_modules ##

a) package.json

package.json is the main configuration file of a Node project.

It contains: 
1) project information, 
2) scripts, 
3) dependencies, 
4) versions of packages your project needs

Purpose: Defines what packages your project needs.



b) package-lock.json

package-lock.json is automatically generated by npm.
It records the exact versions of every dependency installed.

Purpose: 
a) Ensures everyone installs the same versions
b) Makes builds reproducible

c) node_modules
node_modules is a folder where all installed packages live.

Purpose: 
Contains the actual code of dependencies

# How they work together

Flow:

package.json
    ↓
npm install
    ↓
node_modules created
    ↓
package-lock.json generated

# Meaning:

package.json → tells what packages are needed

package-lock.json → locks exact versions

node_modules → stores actual installed packages

## GIT
commit = > package.json, package-lock.json
ignore => node_modules

| File                | Purpose                     |
| ------------------- | --------------------------- |
| `package.json`      | project info + dependencies |
| `package-lock.json` | exact dependency versions   |
| `node_modules`      | installed packages          |


## if change in package.json:

Yes — but not automatically just by editing package.json. You must run npm commands for the changes to apply.

Let’s break it down.

1️⃣ If you only change package.json

Example change:

{
  "dependencies": {
    "react": "^18.2.0"
  }
}

to

{
  "dependencies": {
    "react": "^19.0.0"
  }
}

If you only edit the file, then:

File	Changes?
package.json	✅ yes
package-lock.json	❌ no
node_modules	❌ no

Nothing updates yet.

2️⃣ After running npm install

When you run:

npm install

Then npm will:

Install the new version

Update node_modules

Update package-lock.json

Result:

File	Changes?
package.json	already changed
package-lock.json	✅ updated
node_modules	✅ packages updated
3️⃣ What actually happens internally

Flow:

package.json (version changed)
        ↓
npm install
        ↓
npm resolves dependency tree
        ↓
node_modules updated
        ↓
package-lock.json updated with exact versions
4️⃣ Example with a real package

Suppose you're using:

"lodash": "^4.17.20"

Package:

Lodash

After running:

npm install

npm might install:

lodash 4.17.21

Then package-lock.json will store exactly:

4.17.21
5️⃣ Important rule

👉 node_modules always follows package-lock.json, not directly package.json.

package-lock.json is the source of truth for exact versions.

6️⃣ Special case: npm update

Running:

npm update

Updates packages within the allowed version range defined in package.json.

✅ Final Answer

Yes — when the version in package.json changes and you run npm install:

package-lock.json updates

node_modules installs the new package version

Your project now uses the updated dependency.

== Yes — you should commit package-lock.json to Git when using npm. Even though package.json lists dependencies, package-lock.json is important for consistency and reliability.


What happens during npm install

When you run:

npm install

npm will follow this order:

1. Read package-lock.json
2. Install exact versions
3. Populate node_modules

Not just package.json.

## Project Setup with Parcel

Install Parcel

npm install -D parcel

Run the project: 
npx parcel index.html

Optional scripts in package.json:
"scripts": {
  "start": "parcel index.html",
  "build": "parcel build index.html"
}

Run with: npm start

## Important package.json Rule

Do not include main field in React application.

❌ Wrong

"main": "App.js"

Reason:
main is used for libraries, not applications.

Parcel will throw error:

Library targets are not supported in serve mode

✅ Correct

Remove the main field.
